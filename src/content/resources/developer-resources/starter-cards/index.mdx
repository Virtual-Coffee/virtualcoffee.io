### Starting a new project

Before you write a line of code:

1. Define the goal in one sentence. What does this do, and for whom?
2. Sketch the simplest version, not the full vision, the first working version.
3. Set up version control (Git) before you start.
4. Create a README with the project name, what it does, and how to run it.
5. Pick your stack and commit to it. Don't optimize before you've built anything.

**When you're stuck getting started:** Write the smallest possible thing that proves the concept works. One component, one function, one endpoint. Then build from there.

### Making your first open source contribution

A checklist for contributing to a project for the first time:

1. Read the README and CONTRIBUTING guide before doing anything else.
2. Look for issues labeled `good first issue` or `help wanted`.
3. Comment on the issue before starting work, ask if it's still open and if you can take it.
4. Fork the repo and create a branch with a descriptive name.
5. Make the smallest change that addresses the issue.
6. Write a clear PR description: what you changed, why, and how to test it.
7. Respond to review feedback promptly and politely.

**The most common mistake:** Starting work without checking if the issue is already being worked on.

See also: <Link href="/resources/developer-resources/open-source/contributor-guide">Contributor Guide</Link>

### Asking for help effectively

When you're stuck and need to ask for help:

1. Try to solve it yourself first, spend 20-30 minutes before asking.
2. Describe what you're trying to do, not just what's broken.
3. Include the relevant code, error message, and what you've already tried.
4. Share your environment: language version, framework, OS if relevant.
5. Ask in the right place, a specific Slack channel, a GitHub issue, Stack Overflow.

**The rubber duck test:** Explain the problem out loud to yourself (or an imaginary rubber duck) before asking. You'll solve it more often than you expect.

See also: <Link href="/resources/developer-resources/developer-tips/asking-coding-questions">Asking Better Coding Questions</Link>

### Setting up a new development environment

When starting fresh on a machine:

1. Install a version manager for your primary language (nvm for Node, pyenv for Python, rbenv for Ruby).
2. Set up your editor: install your preferred extensions, configure formatting on save.
3. Set up Git: configure your name, email, and default branch name (`git config --global`).
4. Add an SSH key to GitHub or GitLab.
5. Create a `~/projects` (or similar) directory to keep your work organized.
6. Document what you installed, your future self will thank you.

**Worth doing once:** Create a dotfiles repo to store your config files so setup is faster next time.

### Writing a good commit message

A commit message should tell the story of why a change was made:

- **First line:** Short summary (50 characters or fewer), imperative tense ("Fix login bug" not "Fixed login bug")
- **Body (optional):** Explain the why, not the what. What problem does this solve? What was wrong before?
- **Keep commits small and focused.** One logical change per commit.

**Good:** `Fix null pointer exception when user has no profile photo`

**Less helpful:** `stuff` or `fix` or `updates`

Commit messages are for the person who reads your code six months from now, including you.

### Preparing for a technical interview

A focused checklist for the week before an interview:

1. Review the job description and note the specific technologies mentioned.
2. Practice explaining your past projects out loud in 2-3 minutes each.
3. Do a few coding problems in the format the interview uses (whiteboard, online editor, etc.).
4. Review the fundamentals for your primary language: data structures, algorithms, common patterns.
5. Prepare 3-4 questions to ask the interviewer.
6. Check your setup: audio, camera, screen sharing if it's remote.

**The night before:** Get enough sleep. You'll think more clearly than if you stayed up reviewing notes.

### Writing a good README

A README is the front door of your project. At minimum, include:

1. **Project name and description**, what it does and why it exists
2. **How to install and run it**, exact commands, not summaries
3. **How to use it**, a quick example or screenshot
4. **How to contribute**, even if it's just "open an issue"
5. **License**, choose one and include it

**Optional but useful:**

- Badges (build status, test coverage)
- A demo link or screenshot
- Known issues or limitations
- Links to related docs

A README that's 80% complete and accurate is better than a comprehensive one that's never written.

See also: <Link href="/resources/developer-resources/open-source/maintainer-guide">Open Source Maintainer Guide</Link>

### Navigating burnout

Signs you might be burning out:

- Work that used to feel energizing now feels heavy or pointless
- You're less productive despite working more hours
- You're more irritable than usual
- You've stopped doing things you normally enjoy

**When you recognize it:**

1. Acknowledge it rather than pushing through, burnout that goes unaddressed gets worse.
2. Reduce your load where you can. Something has to give.
3. Protect your off-time. Not answering Slack at 10pm is okay.
4. Do something that has nothing to do with code.
5. Talk to someone, a friend, a therapist, a doctor. You don't have to figure it out alone.

**It's not a character flaw.** Burnout is a response to sustained stress. It happens to capable, hardworking people.

See also: <Link href="/resources/developer-resources/developer-health/burnout">Burnout Resources</Link>

### Giving a lightning talk

A lightning talk is 5 minutes. That's about 500-700 words. You have one job: make one point well.

1. **Pick one idea.** Not an overview, one thing your audience will remember.
2. **Structure it simply:** Hook, context, the thing, takeaway.
3. **Create minimal slides.** One idea per slide. Large text. No walls of text.
4. **Practice out loud** at least twice before you give it.
5. **Don't apologize.** Not for being nervous, not for the talk being simple.

**The most common mistake:** Trying to cover too much in 5 minutes.

See also: <Link href="/resources/developer-resources/public-speaking">Public Speaking Resources</Link> and the [Virtual Coffee Lightning Talk Guide](/assets/pdfs/lightning-talk-guide.pdf)
